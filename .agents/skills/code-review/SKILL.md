# Code review — чек-лист plan-app

Прогоняем по этому списку любой PR **до** approval.

---

## 1. Соответствие плану

- [ ] PR делает только то, что описано в `docs/ROADMAP.md` для этой фазы
- [ ] Не лезем в чужие фазы (например, в Phase 1 PR нет AI-кода)
- [ ] Если расширили scope — это явно отмечено в описании PR + добавлено в `docs/PROGRESS.md`

## 2. Архитектура

- [ ] Не нарушена слоёвость: `app/bot/` не вызывает `app/api/` напрямую (через сервисы из `app/shared/`)
- [ ] Нет круговых импортов (проверка: `python -c "import app"`)
- [ ] Если добавили зависимость в `pyproject.toml` — есть оправдание (зачем именно она)
- [ ] Конфигурация через `app/shared/config.py` (Pydantic Settings), не через `os.environ` в логике

## 3. БД и миграции

- [ ] Любая правка `app/db/models.py` сопровождается Alembic-миграцией (`alembic revision --autogenerate`)
- [ ] Миграция проверена локально на пустой БД (`alembic upgrade head`)
- [ ] Для удаления колонки — двухэтапная миграция (deprecate → remove в следующем PR)
- [ ] Нет N+1 (использован `selectinload` / `joinedload` где нужно)

## 4. AI-код (`app/ai/`)

- [ ] Промпт лежит отдельно в `app/ai/prompts/<name>.md`, не инлайн в коде
- [ ] К промпту есть eval `app/ai/prompts/<name>.eval.yaml` (или отмечено «TODO: eval» с issue)
- [ ] Использован `instructor` для structured output (не самопал на JSON)
- [ ] Все LLM-вызовы обёрнуты в retry + timeout
- [ ] Новый промпт прошёл eval с метрикой ≥ baseline

## 5. Безопасность

- [ ] Никаких секретов в коде (.env / Settings)
- [ ] Никаких `print()` с PII (логируется только `user_id`, не текст сообщения)
- [ ] Webhook валидируется секретным токеном (`X-Telegram-Bot-Api-Secret-Token`)
- [ ] Все user-input проходят через Pydantic-схему (защита от injection)

## 6. Качество кода

- [ ] `uv run ruff format .` — clean
- [ ] `uv run ruff check .` — clean
- [ ] `uv run pytest -q` — all green
- [ ] Никаких `Any`, `getattr`, `setattr` для обхода типизации
- [ ] Нет закомментированного кода («он нам ещё пригодится» — нет, не пригодится)
- [ ] Нет `print()` для отладки (используем `logger`)

## 7. Тесты

- [ ] Любая новая логика покрыта unit-тестом
- [ ] Любой baseline AI-промпт имеет smoke-тест (что хоть что-то возвращает без падений)
- [ ] Тест НЕ требует реального Groq API (mocking через `respx` / `responses`)

## 8. Документация

- [ ] Если изменили публичное поведение — обновлён `docs/PROGRESS.md`
- [ ] Если изменили архитектуру — обновлён `docs/ARCHITECTURE.md`
- [ ] Если появилась новая идея на потом — добавлено в `docs/IDEAS.md`
- [ ] PR-описание объясняет «зачем», не только «что» (см. `.agents/skills/writing-prs/SKILL.md`)

## 9. UX (если касается бота / мини-аппки)

- [ ] Сообщения — на русском, человеческим языком
- [ ] Длинные сообщения разбиты на читаемые блоки (≤ 4096 символов в Telegram)
- [ ] Inline-клавиатуры используются вместо текстовых команд где можно
- [ ] Ошибки показаны юзеру, а не молча

## 10. Перформанс

- [ ] Webhook-handler возвращает 200 за < 1 сек (тяжёлая работа — в очереди)
- [ ] LLM-вызовы — асинхронные (`asyncio.gather` для параллельных)
- [ ] Запросы в БД пакетные (не цикл с одиночными `commit`)

---

## Когда блокировать PR

- ❌ Падает CI / ruff / pytest
- ❌ Нет миграции при правке моделей
- ❌ Секреты в коде
- ❌ AI-промпт без eval
- ❌ Промпт инлайн вместо `prompts/`

## Когда писать комментарий, но не блокировать

- 💬 «Можно бы переименовать переменную для ясности»
- 💬 «Тут можно бы добавить кейс в тесте, но не критично»
- 💬 «Альтернативный путь: …»

## Когда быстро аппрувить

- ✅ Чисто Phase 0/1-уровня (правки доков, конфигов, рефакторы без логики)
- ✅ Узкий, понятный bugfix с тестом
