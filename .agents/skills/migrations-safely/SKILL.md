# Migrations safely (Alembic + SQLModel)

plan-app uses Alembic against SQLModel models. Migrations live in `alembic/versions/`. Render's Postgres has the prod schema — you can't `drop_all` it without losing data.

---

## 1. Iron rule

**Any change in `app/db/models.py` ships with an Alembic migration in the same PR.** If it's not in the same PR, the deploy will fail or silently diverge from the model.

---

## 2. Generate a migration

```bash
# 1. edit app/db/models.py
# 2. autogenerate
uv run alembic revision --autogenerate -m "phase X: <short-slug>"
# 3. open alembic/versions/<NEW>.py and READ it
# 4. apply locally on a fresh SQLite DB
uv run alembic upgrade head
# 5. apply on Postgres if you have one (optional but ideal)
```

Naming: `NNNN_<phase>_<slug>.py`, e.g. `0003_phase_4_reminders.py`. Numbers strictly monotonic.

---

## 3. Read the autogenerated diff before committing

Autogenerate is **suggestive, not authoritative**. Common false positives:

| Symptom in the migration | What's actually happening |
|---|---|
| `op.drop_column('users', 'tz')` you didn't intend | A model field was renamed but Alembic saw drop+add. Fix manually with `op.alter_column(..., new_column_name=...)`. |
| `op.create_index(...)` on every run | SQLModel's implicit indexes drift between revisions. Use `index=True` on the model field once, then ignore re-creation suggestions. |
| Empty `upgrade()` / `downgrade()` | You ran autogenerate on a clean DB. Run `alembic upgrade head` first to sync, then autogenerate. |

---

## 4. Safe column drop — two-step

Postgres locks the table while `DROP COLUMN` runs. For prod safety:

1. **PR N:** stop writing to the column. App ignores it. Migration is empty.
2. **PR N+1:** `op.drop_column(...)`.

For plan-app at one user, single-step drop is fine. Document the risk in the PR description.

---

## 5. Adding a non-nullable column

```python
# WRONG — breaks existing rows
op.add_column("tasks", sa.Column("priority", sa.Integer(), nullable=False))

# RIGHT — nullable + server_default first, then optionally tighten later
op.add_column(
    "tasks",
    sa.Column("priority", sa.Integer(), nullable=False, server_default="0"),
)
```

`server_default` populates existing rows. After the migration, you can drop the default in app code if you don't want it persisted.

---

## 6. JSON columns

We use SQLAlchemy's generic `JSON` (`from sqlalchemy import JSON`) — works on SQLite (TEXT) and Postgres (JSONB). For **structured** data prefer JSONB-only with `sa.dialects.postgresql.JSONB` and gate on dialect — but only when you actually need JSONB operators.

Don't autogenerate `op.alter_column(... type_=sa.JSON())` on every revision — `render_as_batch=True` in `env.py` already handles SQLite's lack of `ALTER COLUMN`.

---

## 7. Naive-UTC `DateTime`

All datetime columns in plan-app are `DateTime(timezone=False)` (the default for SQLModel). Writers must use `app/shared/time.py::utcnow_naive()`. See `.agents/skills/defensive-programming/SKILL.md::3`.

If you ever add a tz-aware column, do it explicitly (`sa.DateTime(timezone=True)`) and document why.

---

## 8. Local testing

```bash
# Wipe the SQLite test DB and replay all migrations from scratch
rm -f .data/dev.db
uv run alembic upgrade head

# Round-trip: upgrade then downgrade then upgrade
uv run alembic downgrade -1
uv run alembic upgrade head
```

If `downgrade -1` raises, the migration is **broken** even if `upgrade` works. Fix it.

---

## 9. Production deploy

Render runs `uv run alembic upgrade head` on every deploy (see `Dockerfile` / `render.yaml`). To roll back:

1. Identify the bad migration revision.
2. Locally: `uv run alembic downgrade <good-rev>`.
3. Run the **inverse** SQL on prod via Render Shell (Render Free has no shell — use a one-off script).
4. Revert the PR, push, redeploy.

There's no "undo deploy" button.

---

## 10. Anti-patterns

- ❌ Ad-hoc SQL via `session.execute("ALTER TABLE …")` — bypasses Alembic, won't replay.
- ❌ Editing an existing merged migration — you have no idea who already ran it.
- ❌ Multiple unrelated schema changes in one revision — split.
- ❌ Forgetting to commit `alembic/versions/<NEW>.py` to the PR.
