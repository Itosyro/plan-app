# Russian NLP — выжимка для plan-app

Русский язык — это **флективная морфология** + свободный порядок слов + контекстные омонимы. Игнорировать = терять качество.

---

## 1. Парсинг дат / времени

### Стек
- **`dateparser`** — основная либа. Понимает «через 43 минуты», «во вторник», «на следующей неделе», «вчера».
- Fallback: regex для совсем простых кейсов (`(\d+)\s*(минут|часов|дней)`).

### Конфиг
```python
import dateparser

settings = {
    "PREFER_DATES_FROM": "future",   # "завтра" → завтра, не вчера
    "TIMEZONE": user_tz,             # tz юзера, не сервера
    "RETURN_AS_TIMEZONE_AWARE": True,
    "RELATIVE_BASE": datetime.now(user_tz),
    "DATE_ORDER": "DMY",             # "01.05.2026" → 1 мая, не 5 января
}

dt = dateparser.parse("через 43 минуты", languages=["ru"], settings=settings)
```

### Грабли

| Кейс | Что dateparser ломает | Решение |
|---|---|---|
| «во вторник» (если сегодня вторник) | парсит как сегодня | проверить `dt.date() == today` → +7 дней (next вторник) |
| «вечером» | возвращает 18:00 (default) | подмена в препроцессе на «в 19:00» |
| «утром» | 9:00 | подмена на «в 09:00» |
| «через час» | работает корректно | OK |
| «на этой неделе» | возвращает текущий момент (хм) | препроцесс → «в воскресенье 23:59» (см. `docs/ARCHITECTURE.md` §2.4) |
| «в следующий понедельник» | не понимает «следующий» иногда | замена «следующий»→«next» перед `dateparser` |

### Ru-специфика
- «через **полчаса**» — заменить на «через 30 минут» в препроцессе
- «через **полтора часа**» — заменить на «через 1 час 30 минут»
- «через **сутки**» — заменить на «через 24 часа»
- «**к** утру» — заменить на «в 9:00»

См. `app/ai/datetime_pre.py` (когда напишем) — там нормализация перед `dateparser`.

## 2. Лемматизация (для матчинга категорий)

### Зачем
Юзер пишет «пробежка», в БД категория «бег». Нужно понять, что это одно. Лемма обоих → «бег».

### Стек
- **`pymorphy3`** (форк pymorphy2 с поддержкой Python 3.12+)

```python
import pymorphy3

morph = pymorphy3.MorphAnalyzer()
lemma = morph.parse("пробежка")[0].normal_form  # → "пробежка" (увы, разные основы)
lemma = morph.parse("бегу")[0].normal_form     # → "бежать"
```

### Грабли
- pymorphy для **существительного** даёт основу-существительное; для **глагола** — инфинитив. Их сравнение — это уже не строгий match. Использовать **fuzzy** (rapidfuzz) на уровне ≥ 0.7 как backup.
- Слова с одинаковой леммой могут иметь разные значения («коса» — волосы / орудие / география). Контекста pymorphy не учитывает.

### Стратегия в plan-app
1. lower-case + strip
2. лемматизация всех значимых слов входа (части речи NOUN, VERB, INFN, ADJF)
3. лемматизация ключевых слов категории
4. **set-пересечение** + минимум 1 общее слово → match
5. иначе — спросить LLM-классификатор

## 3. Токенизация на предложения

`razdel` — лучшая по качеству, не зависит от nltk-моделей.

```python
from razdel import sentenize, tokenize

for s in sentenize("Сегодня в 9 пробежка. Вечером — встреча с Машей."):
    print(s.text)
```

Используется в Splitter перед раздачей в Classifier.

---

## 4. Pipeline по тексту юзера

```
голос (voice) → Whisper (Groq) → text
text → razdel.sentenize() → [sent1, sent2, …]
[sent_i] → AI Splitter → [intent_i, …]
intent_i → datetime_pre.normalize() → text_norm
text_norm → AI Classifier → структура
              ↓
              dateparser (если есть time-фраза)
              pymorphy3 (если есть категория)
```

## 5. Что НЕ делать

- ❌ Использовать `nltk` без скачивания russian models — не работает
- ❌ Регулярки на даты вида `\d+ \w+` без `dateparser` — слишком много false positives
- ❌ Ленивая лемматизация (без правил по частям речи) — мусор в категориях
- ❌ Хранить в БД сырые формы слов — только леммы (для индекса)
