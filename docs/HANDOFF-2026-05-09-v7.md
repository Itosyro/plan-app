### I-1: defensive callback parsing (`7bab98d`)

**Проблема:** 7 callback handlers в `app/bot/routers/callbacks.py` делали
`int(parts[N])` без `try/except`, так что любой битый или
злоумышленный payload (`task:done:abc`, `task:done:`,
`task:done:1:2:3`) роняет хендлер с `ValueError` → Telegram видит
500 → retry → spam в логах.

**Фикс:** общий парсер `parse_task_callback(data, action, *, arity)`
в `callbacks.py`. Возвращает `(task_id, extra_parts)` или `None`.
Все 7 хендлеров (`task:done`, `task:pick_move`, `task:move`,
`task:delete`, `task:pick_category`, `task:set_category`,
`task:cancel`) теперь делают `if parsed is None: return ...` плюс
`callback.answer("Неверный формат.")`.

**Тесты:** 7 кейсов в `test_callbacks.py` (валидные/битые/short/long
arity, non-numeric task_id, неверный action prefix).

### I-2: race-safe upsert via ON CONFLICT (`e6d9a6d` + `82ae938`)

**Проблема:** `get_or_create_category` и `get_or_create_horizon`
делали SELECT, потом INSERT. Под parallel `asyncio.gather`
(пять одновременных classify в одном split) две корутины
одновременно делали SELECT (пусто) → две INSERT → одна падает на
UNIQUE (`user_id`, `name`).

**Фикс:** `INSERT ... ON CONFLICT DO NOTHING` через dialect-aware
`Insert` (`_PgInsert | _SqliteInsert`), потом `flush + re-SELECT`.
ORM-bypassing Core SQL. Работает в Postgres и SQLite.
Followup-коммит `82ae938`: typing union для `Insert` + ruff C416 fix
(set comprehension → `set()`).

**Тесты:** 5 кейсов в `test_persistence.py` — concurrent gather,
повторный вызов получает существующую row, race не дублирует и т.д.

### I-3: idempotent /start (`bb65899`)

**Проблема:** `complete_onboarding` всегда делал
`session.add(UserSettings(user_id=user.id))`. Если пользователь
сделал `/start` второй раз (после wipe чата) — `IntegrityError` на
`UserSettings.user_id` PK.

**Фикс:** SELECT первым. Если row существует — оставляем. Только
обновляем User (`display_name`, `tz`, `onboarded_at`).
Пользовательские настройки (`critic_mode`, `morning_digest_at`)
сохраняются через re-onboarding.

**Тесты:** 1 кейс в `test_services.py` — сетит settings,
re-onboardit, проверяет что settings не сбросились + только одна
row.

### I-4: digest catch-up (`656a878`)

**Проблема:** `tick_digests` использовал strict-minute match
`local.hour == hh and local.minute == mm`. Render scheduler-tick
может прийти в `08:00:00`, потом в `08:01:30` (cold-start, GC
pause). Минутный матч → дайджест на день потерян. Существующий
`last_*_digest_on` гард только предотвращал DOUBLE-send, не
помогал с пропущенным.

**Фикс:** catch-up — fire когда `local_now >= scheduled_time` и
`last_*_digest_on != today`. Плюс day-1 safeguard: если юзер
закончил онбординг *после* сегодняшнего slot-time — первый
дайджест этого типа приходит завтра, не сейчас. Иначе юзер с
онбордингом в 21:00 и `morning_digest_at=08:00` мгновенно
получает «доброе утро».

**Тесты:** 6 новых в `test_digest.py`. Один _onboard helper
обновлён, чтобы существующие тесты с фиксированным
`now=2026-05-08` не триггерили day-1-suppression — установлен
дефолтный `onboarded_at = 2026-04-28`.

### I-5: claim-pattern + per-row commit (`f542ef4`)

**Проблема:** `tick_reminders` собирал batch, мутировал status в
памяти, коммитил в конце. SIGTERM mid-batch — все sent-rows
откатываются → следующий tick шлёт duplicates.

**Фикс:** трёхступенчатый state machine.
1. **Claim:** атомарный
   `UPDATE reminders SET status='processing' WHERE id=:id AND status='pending'`
   + commit. Если `rowcount != 1` — другой воркер забрал, skip.
2. **Send:** Telegram round-trip.
3. **Terminal flip:** `status='sent'`/`'failed'`/`'pending'`
   (revert на retry) + commit per-row.

Stuck `processing`-rows (kill между claim и flip) — оставляем
оператору. Без `claimed_at` колонки автоматически decision'ить
unsafe (нельзя сказать была ли доставка).

**Тесты:** 3 регрессии в `test_scheduler.py` — два tick'а на одной
cutoff не дублирует, processing-row не подбирается, per-row commit
изолирует крах в середине batch'а.

### I-6: /today single-message + compact keyboard (`f6f5c72`)

**Проблема:** `_horizon_handler` шлёт 1 summary + N task messages
с каждой своей 4-кнопочной клавиатурой. 30 задач → 31 telegram
message. Spam.

**Фикс:** один message per call. Summary с задачами 1..N + одна
inline-keyboard с 4 emoji-only кнопок per row (`N ✅ N 🔄 N 🗑 N 🏷`).
Page cap = 20 задач (80 кнопок < 100 лимит Telegram). Если задач
больше — overflow line «Показано 20 из N. Используй /search или
фильтр по категории». Callback_data unchanged
(`task:done:<id>` etc.) — все existing хендлеры работают.

Новый helper `horizon_list_keyboard()` в `callbacks.py`. Старый
`task_action_keyboard()` сохранён (используется в re-render
после `task:cancel`).

**Тесты:** 4 новых в `test_commands.py` — overflow message
present/absent, keyboard layout (rows × buttons), page-size
guard против будущего раздувания.

### I-8: pipeline backpressure (`9ccf7ea`)

**Проблема:** `run_pipeline` запускался через
`asyncio.create_task(_background())` без backpressure. Spam-tap
voice → unbounded fan-out, Groq quota race, file-descriptor
exhaustion на shared client pool.

**Фикс:** два semaphores в `_pipeline.py`.
* **Per-user** (`PER_USER_PIPELINE_LIMIT = 1`): строгая
  сериализация. Lock per `user_id` через
  `asyncio.Lock`-protected dict.
* **Global** (`GLOBAL_PIPELINE_LIMIT = 8`): cap на total
  concurrency. Нужен headroom на single Render instance.

Acquisitions nested: per-user *outside* global → не может быть
deadlock'а где все global-slot'ы держат юзеры ждущие global.
Pipeline body вынесен в `_run_pipeline_inner` для testability
(monkeypatch).

**Тесты:** 7 кейсов в новом `test_pipeline_backpressure.py` —
limit constants, per-user vs global identity,
serialisation order, global cap enforcement, multi-user
parallelism, fixture для reset cached semaphores между тестами
(каждый тест свой event loop).

## Грабли, которые встретил

### G-1: SAVEPOINT не работает с asyncio.gather

Изначально пытался I-2 решить через `async with session.begin_nested():
INSERT; commit;` + ловить `IntegrityError`. Под parallel
`asyncio.gather` SAVEPOINT не reliable: `session.expunge(new_cat)`
кидал `InvalidRequestError` потому что объект уже detached.
Бросил, перешёл на `INSERT ... ON CONFLICT DO NOTHING`.

### G-2: mypy `Insert` has no `on_conflict_do_nothing`

Generic `_Insert` хинт не expose'ит dialect-specific методы.
Решил через union `_PgInsert | _SqliteInsert`.

### G-3: existing digest tests падают после I-4

Шесть тестов в `test_digest.py` и один в `test_e2e_phase4.py`
ожидали strict-minute match. После catch-up они не fail'ятся
напрямую но fall through к day-1 safeguard который suppress'ит
дайджест (потому что `_onboard()` ставил `onboarded_at = utcnow`,
а тесты использовали `now=2026-05-08`).

Решение: дать `_onboard()` параметр `onboarded_at` с дефолтом
`datetime(2026, 4, 28)` — 10 дней до тестовой даты — чтобы
day-1 safeguard не срабатывал в legacy-тестах. Новые тесты для
day-1 safeguard передают свежий `onboarded_at` явно.

### G-4: mypy `Result[Any]` no `rowcount`

`session.execute(update(...))` возвращает union которое mypy
widens к `Result[Any]`. `.rowcount` есть только на
`CursorResult`. Решил через `assert isinstance(claim_result,
CursorResult)`. `getattr` запрещён `defensive-programming/SKILL.md`.

### G-5: asyncio.Semaphore в test fixture

При reuse между тестами (один и тот же loop-binding) семафор
ловит CancelledError. Сделал `reset_pipeline_semaphores_for_tests()`
test-only hook + autouse fixture в `test_pipeline_backpressure.py`.
