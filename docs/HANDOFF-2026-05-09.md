## 1. Что это за проект

### 1.1. Основная идея

Юзер пишет/говорит «поток сознания» в Telegram. Бот:
1. **Транскрибирует** голос (Whisper-large-v3).
2. **Сплитит** на отдельные интенты — Splitter (`llama-3.1-8b-instant`).
3. **Классифицирует** каждый интент: категория / горизонт / приоритет / задача-vs-заметка — Classifier (`llama-3.3-70b-versatile`).
4. **Критикует** результат при низкой уверенности — Critic (`qwen-qwq-32b`).
5. **Резолвит время** — `dateparser` + `pymorphy3` + `razdel` (Pure Python, без LLM).
6. **Сохраняет** в БД (Task / Note / AiRun / TaskEvent) и **планирует** напоминания (Reminder, по умолчанию `[60, 15]` минут до due_at).
7. **Отвечает** через Courier — 50/50 шаблон / `llama-3.1-8b-instant` (источник и тон управляются настройками юзера).

Параллельно in-process scheduler (раз в 60с) тикает:
- **`tick_reminders`** — отправляет напоминания, ретраит до 3 раз с exponential backoff, после `MAX_REMINDER_ATTEMPTS=3` ставит `status='failed'`.
- **`tick_digests`** — морнинг/ивнинг дайджесты по `UserSettings.morning_digest_at` / `evening_digest_at`, гвардятся `last_*_digest_on` чтобы не задвоить.

**Коммит `6ca9d41` — C-2: `Task.due_at` UTC normalisation + `format_due_local` helper**

Проблема:
- `dateparser.parse("завтра в 15:00", ..., settings={"TIMEZONE": user.tz, "RETURN_AS_TIMEZONE_AWARE": True})` возвращает **tz-aware MSK** datetime.
- `Task.due_at: datetime | None = Field(...)` — **tz-naive колонка**.
- При вставке `Task(due_at=aware_msk)` SQLAlchemy/SQLModel **молча дропает tzinfo** — в БД оказывается naive значение **с MSK clock-time**, нарушая schema contract «naive == UTC».
- `Reminder.fire_at` уже был защищён через `_to_naive_utc` в `schedule_reminders`, но `Task.due_at` — нет.
- Симптом был **замаскирован**: все 3 display-сайта (`workers/scheduler.py::_format_reminder`, `bot/digest.py::_format_task_line`, `bot/routers/commands.py::_format_task_list`) форматировали `due_at` через `:%H:%M` напрямую, что *случайно* совпадало с тем что юзер сказал.
- **Скрытый баг ждал**: любое сравнение `now_utc < task.due_at` или JSON-export для Phase 5 mini-app был бы off by tz_offset.

Решение:
- **Новый `app/shared/time.py::to_naive_utc(dt) -> datetime`** — публичная версия того, что было в `services._to_naive_utc`. Tz-aware → UTC → strip tz; tz-naive → returned as-is. Алиас `_to_naive_utc = to_naive_utc` оставлен для back-compat.
- **Новый `app/shared/time.py::format_due_local(due_at, user_tz) -> str | None`** — единственный правильный способ отрендерить naive-UTC datetime в HH:MM в локальной TZ юзера. Возвращает `None` для local-midnight (sentinel «date-only deadline»). Graceful fallback на UTC при невалидной TZ.
- `app/bot/services.py::persist_classification` — теперь `due_at_utc = to_naive_utc(due_at) if due_at is not None else None` перед `Task(...)`.
- `app/workers/scheduler.py::_format_reminder(task, user_tz)` — берёт `user_tz` параметром, использует `format_due_local`. Call-site в `tick_reminders` передаёт `user.tz`.
- `app/bot/digest.py::_format_task_line(task, user_tz)` — то же. Call-sites в `build_morning_digest` и `build_evening_digest` передают `user.tz`.
- `app/bot/routers/commands.py::_format_task_list(tasks, title, user_tz)` — то же. Call-site в `_horizon_handler` берёт `user_tz = user.tz` до выхода из `session_scope` (потому что `user` после выхода detached).
- Тесты:
  - **Новый файл `tests/test_shared_time.py`** (8 тестов): round-trip aware-MSK → naive-UTC, naive-input passthrough, format_due_local в разных TZ (Moscow, UTC), local-midnight → `None`, graceful fallback на невалидную TZ, smoke-тест `utcnow_naive`.
  - `tests/test_persistence.py` — 3 новых теста (`test_persist_classification_normalises_aware_due_to_naive_utc`, `_naive_due_at_passes_through`, `_none_due_at_stays_none`).
  - Обновлены `tests/test_scheduler.py::test_format_reminder_*` (теперь принимают `user_tz`, проверяют что 11:00 UTC в MSK = 14:00) и `tests/test_commands.py::test_format_task_list_with_tasks` (12:30 UTC в MSK = 15:30).

Результат: 180 → 193 passed. Без миграции (схема не менялась — только семантика записи).

**Коммит `f647415` — C-3: digest double-send guard + `last_*_digest_on` колонки**

Проблема:
- `tick_digests` в основной цикле сравнивал `_user_local_now(user.tz, now_utc=now)` с `settings.morning_digest_at` / `evening_digest_at` через `_matches_hhmm` (строгое равенство HH:MM).
- При `SCHEDULER_TICK_INTERVAL_SECONDS < 60` (например, 10с) внутри минуты совпадения было **6 тиков** — каждый отправлял дайджест.
- У `Reminder` есть `status='pending' → 'sent'` гвард, у дайджестов — нет.
- На прод сейчас `interval=60`, симптом не наблюдался; но фикс защищает от любого сокращения интервала в будущем (например, при переезде scheduler'а в Phase 5 mini-app).

Решение:
- `app/db/models.py::UserSettings` — две новые колонки: `last_morning_digest_on: date | None` и `last_evening_digest_on: date | None`. Хранят **user-local дату**, на которую отправлялся последний digest.
- `app/bot/digest.py::tick_digests` — перед отправкой сверяет `settings.last_morning_digest_on == local.date()` (и evening), если совпадает — пропускает. После успешной отправки выставляет колонку и `await session.flush()` чтобы гвард был виден последующим итерациям цикла внутри одного `session_scope`.
- **Alembic 0005** (`2026_05_09_0700-0005_digest_idempotency_guards.py`) — добавляет две `date NULL` колонки. Существующие пользователи получают `NULL` → первый тик в день после миграции отправит как обычно, дубль не случится.
- Тесты в `tests/test_digest.py`:
  - `test_tick_digests_skips_second_call_in_same_minute` — два тика при том же `now` шлют ровно один morning.
  - `test_tick_digests_skips_repeat_within_same_minute_evening` — то же для evening.
  - `test_tick_digests_resends_on_next_day` — гвард по дате: завтра в то же HH:MM отправит.
  - `test_tick_digests_persists_gate_to_db` — колонка реально коммитится `session_scope`.

### 4.2. Секреты

|---|---|---|

| `TELEGRAM_WEBHOOK_SECRET_PATH` | Случайная строка в URL `/tg/<secret>` | uuidgen / openssl rand |

| `GROQ_API_KEYS` | Comma-separated список ключей (для round-robin) | https://console.groq.com/keys |

| `SENTRY_DSN` | optional | https://sentry.io |

Local dev: создай `.env` в корне (gitignored), скопируй из `.env.example`. Для тестов секреты не нужны — все Groq-вызовы замоканы через `respx`.

### 4.3. Git авторизация и workflow

- **Девин-прокси не видит репо `Itosyro/plan-app`** (он не в whitelist), поэтому push нужно делать с PAT.
- В этой сессии использовался **временный PAT** через `secrets` систему: `GITHUB_PAT_ITOSYRO_PLANAPP`.
- Команда push: `git push "https://x-access-token:${GITHUB_PAT_ITOSYRO_PLANAPP}@github.com/Itosyro/plan-app.git" <branch>:<branch>`.
- `git_pr(action="create")` возвращает `Resource not accessible by personal access token` → создавай PR'ы напрямую через GitHub API: `curl -X POST -H "Authorization: token $TOKEN" -d @body.json https://api.github.com/repos/Itosyro/plan-app/pulls`.
- Ветки: `devin/<unix_ts>-<short-name>` (например `devin/1778308687-critical-fixes-c1-c2-c3`).
- **Не амендить коммиты**, **не force-push**, **не push в main**, **не committи `.env`**, **не --no-verify**.

### 4.4. CI / деплой

- GitHub Actions workflow в `.github/workflows/ci.yml` (один job): `uv sync` → `ruff format --check` → `ruff check` → `pytest -q`. Это всё.
- Деплой: Render Free web-service на push в `main`. `render.yaml` декларирует один web-сервис с `Dockerfile`. Alembic миграции **не** автоприменяются — нужно запускать вручную или в `release.command` (см. `docs/RENDER.md`).
- В прод сейчас **в-процессный scheduler**: `app/main.py::lifespan` запускает `app/workers/runner.py::start_inproc_scheduler` параллельно с FastAPI. Free-tier не разрешает отдельный cron-сервис.
