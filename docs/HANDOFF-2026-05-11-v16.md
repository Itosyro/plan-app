### Phase 0..1 — фундамент

- Python 3.12 + uv + FastAPI + aiogram 3 + SQLModel + Alembic.
- Webhook на `/telegram/webhook` (а не polling).
- `/start`, `/help`, миграции, базовый текст→Task без AI.
- Render Free + Neon Postgres работают.

### Phase 2 — AI-пайплайн (PR-1..PR-3 в старом счёте)

- **Splitter** (`app/ai/splitter.py`): `llama-3.1-8b-instant` через instructor → `SplitResult.units[]`. Разбивает «утром пробежка, в 11 совещание, до пятницы отчёт» на 3 единицы.
- **Classifier** (`app/ai/classifier.py`): `llama-3.3-70b-versatile` → `ClassifierResult` (kind: task/note/reminder, category, horizon, priority, due_at_raw, first_step).
- **Time Resolver** (`app/ai/time_resolver.py`): детерминированный (без LLM), `dateparser` + `pymorphy3` для русских падежей. «через 43 минуты», «во вторник», «утром», «к концу недели», «через 2 часа».
- **Critic** (`app/ai/critic.py`): `qwen-qwq-32b` (reasoning) перепроверяет классификатор когда `confidence < threshold` или `paranoid` mode.
- **Whisper** (`app/ai/whisper.py`): `whisper-large-v3` (точная, не turbo — юзер требовал максимум).
- **GroqKeyRouter** (`app/ai/router.py`): round-robin с per-key cooldown'ом на 429. Сейчас 3 ключа из `GROQ_API_KEYS=key1,key2,key3`.

### Phase 2.2 — БД + категории/горизонты

- Модели: `User`, `Category` (per-user), `Horizon` (per-user, slug = today/tomorrow/week/month/year/someday + кастомные), `Task`, `Note`, `InboxEntry`, `TelegramUpdate` (idempotency).
- Авто-создание Category и Horizon в `get_or_create_category` / `get_or_create_horizon`.

### Phase 2.3d — Reorder голосом

- «Перенеси отчёт на пятницу» / «переложи йогу на завтра» / «отчёт — на неделе».
- Pipeline сначала зовёт `_try_reorder` → `detect_reorder` → если интент = reorder → `update_task_horizon` без классификатора.
- Возвращает «Перенёс „X“ → завтра.».
- **Что НЕ умеет:** только меняет horizon (today/tomorrow/week/month/year/someday). Не меняет дедлайн в часах/минутах, не понимает кастомные горизонты, не переносит между категориями.

### Phase 3 — Категории, горизонты, реминдеры (PR-2..PR-4)

- Кастомные горизонты (LLM сам создаёт).
- Реминдеры: `Reminder` модель + извлечение «напомни через X» + дефолтные офсеты (1ч+15мин если задача сегодня, 1д+1ч если через несколько дней).
- Команды `/today`, `/week`, `/notes`, `/settings`, `/cancel`, `/tasks`.

### Phase 4 — Scheduler + дайджесты (PR-?, PR-G, PR-89)

- `app/workers/scheduler.py::tick_reminders` каждую минуту шлёт реминдеры с `due_at <= now`.

- **PR-G (только что):** утренний теперь включает Просрочено + Горячие дедлайны недели; вечерний — Закрыто сегодня + Осталось + Завтра. См. §5 PLAN.md.
- **PR-89 (только что):** keep-alive self-ping `/healthz` каждые 10 мин — Render Free больше не засыпает.

### Phase 5 — Mini App (PR-?, PR-Notes, PR-D, PR-E, PR-UX)

### 5.2 · Intent taxonomy (что мы умеем распознать)

| Intent | Английское имя | Примеры русских формулировок |
|---|---|---|
| **create** | `create` | _Текущий путь, не меняем._ «утром пробежка», «купить хлеб», «напомни через час про обед». |
| **reorder_horizon** | `reorder` | «перенеси отчёт на пятницу», «йогу — на завтра», «положи это в неделю», «это на потом», «перекинь в когда-нибудь». |
| **reorder_time** | `reorder_time` | «перенеси встречу с 14 на 18», «йога не в 7, а в 8», «передвинь стендап на час позже», «давай завтра в 11, а не в 9», «после обеда вместо утра». |
| **complete** | `complete` | «закрой йогу», «отметь пробежку как сделанную», «я сделал отчёт», «йогу закрыл», «убери из активных — я её сделал», «сделано», «done». |
| **delete** | `delete` | «удали йогу», «убери из списка пробежку», «эта задача уже не актуальна, удали», «отмени поход к врачу» (если задача), «выкинь из плана». |
| **reopen** | `reopen` | «верни йогу — я её не сделал», «зря закрыл, верни в активные», «нет, ещё не сделал». |
| **rename** | `rename` | «переименуй „купить хлеб“ в „купить хлеб и молоко“», «исправь название», «переформулируй задачу X». |
| **set_due** | `set_due` | «поставь дедлайн на пятницу 12:00», «дедлайн — до конца недели», «крайний срок — вторник», «горит до завтра». |
| **set_priority** | `set_priority` | «сделай отчёт срочным», «йога — низкий приоритет», «это важно», «не критично», «горячо», «не горит». |
| **set_category** | `set_category` | «перенеси „купить хлеб“ в покупки», «отнеси встречу в работу», «это здоровье, а не работа». |
| **list_completed_today** | `list_done` | «что я закрыл сегодня», «покажи закрытые», «список сделанного сегодня». (Read-only, новая команда.) |
| **none** | `none` | Не интент-команда — обычное создание задачи (текущий путь). |

### 5.3 · NL-вариативность — как юзер реально говорит

> Юзер прямо просил: «ключевые слова перенести изменить, да, поменяй, не все же вот это прям говорят, они немного по-другому говорят свои мысли, вслух говорят в потоке ещё». Поэтому промпт должен ловить **смысл**, а не keyword'ы.

**Перенос времени (`reorder_time` / `set_due`):**

- «перенеси на 8 утра» (явное время)
- «давай не в 3, а в 8» (relative от прежнего)
- «передвинь встречу на час позже» / «на полчаса раньше» (offset)
- «после обеда» (anchor)
- «к концу недели» (deadline-style)
- «не сейчас, после 18» (lower-bound)
- «вместо завтра — пятница» (cross-axis: меняет и horizon, и due_at)
- «давай пораньше» / «давай попозже» (без числа — это **clarification trigger**, бот должен спросить «на сколько раньше?» или предположить 1 час)

**Закрытие задачи (`complete`):**

- «сделал X» / «X сделал» (perfective verb past tense)
- «X готов»
- «X — done»
- «закрой X» / «закрыл X» / «отметь X закрытой»
- «выполнил X»
- «отметь X как сделанную» / «отметь сделанным» (морфология)
- «убери из сегодня — это уже сделано»
- «я её сделал» (anaphora к последней задаче — нужен контекст)

**Удаление (`delete`):**

- «удали X» / «убери X»
- «это уже не нужно, выкинь»
- «отмени X» (если X — задача, не реминдер — `delete`. Если реминдер — отмена реминдера, см. §3-D)
- «не делай эту задачу»
- «такого больше не будет, удали»

**Приоритет (`set_priority`):**

- «срочно» / «горит» / «важно» / «критично» → `high`
- «не срочно» / «не горит» / «можно потом» → `low`
- «обычно» / «средне» → `medium`
- «эту в первую очередь» (high)
- «эту в последнюю» (low)

**Категория (`set_category`):**

- «это работа» / «это здоровье»
- «отнеси к финансам»
- «положи в покупки»
- «это идея, не задача» — на самом деле это `kind_change` (Task→Note), но скорее всего юзер так редко формулирует, чтобы делать отдельный интент.

### 5.4 · Архитектура intent-router'a

Сейчас в `_pipeline.py::run_pipeline` идёт:
```
text → _try_reorder → (if reorder) update + return
     → split_message → for unit: time → classify → critic → persist → reply
```

Предлагаемая архитектура (после PR-I1..I3):

```
text → detect_intent (один LLM-call) → switch:
       case create        → текущий путь (split → classify → ...)
       case reorder_*     → execute_reorder
       case complete      → execute_complete
       case delete        → execute_delete
       case reopen        → execute_reopen
       case rename        → execute_rename
       case set_due       → execute_set_due
       case set_priority  → execute_set_priority
       case set_category  → execute_set_category
       case list_done     → execute_list_done
       case none          → fallback to create-path
```

`detect_intent` возвращает структурированный `EditIntent`:

```python
class EditIntent(BaseModel):
    intent: Literal[
        "create", "reorder_horizon", "reorder_time", "complete",
        "delete", "reopen", "rename", "set_due", "set_priority",
        "set_category", "list_done", "none",
    ]
    task_query: str | None = None    # для всех edit-intent'ов
    new_horizon: Literal["today", "tomorrow", "week", "month", "year", "someday"] | None = None
    new_due_raw: str | None = None   # «пятница 12:00» — потом через time_resolver
    new_title: str | None = None
    new_priority: Literal["high", "medium", "low"] | None = None
    new_category: str | None = None
    confidence: float = 0.0  # 0..1 — для уточнения если < 0.7
```

Один LLM-call вместо двух (текущий reorder + классификатор) — экономит latency и tokens. Модель — `llama-3.1-8b-instant` через instructor.

### 5.5 · Ambiguity и multi-match handling

`find_task_by_query(session, user_id, query)` сейчас возвращает первую совпавшую (LIKE по title). Это плохо для edit:

- Если совпадений **0** — отвечать «Не нашёл задачу „X“. Может, она уже закрыта или ты её удалил?» с inline-кнопкой `Показать архив`.
- Если совпадений **1** — выполнять немедленно.
- Если совпадений **>1** — отвечать «Нашёл несколько подходящих, уточни какую:» + inline-keyboard с топ-5 совпадениями (TaskCard-like кнопки). Callback `edit:resolve:<intent>:<task_id>:<payload_json>`.

### 5.6 · Контекст «последняя задача» (PR-I3)

После любой операции (create/edit) бот хранит `last_task_id` в in-memory dict с TTL=60с:

```python
LAST_TASK: dict[int, tuple[int, float]] = {}  # user_id -> (task_id, expires_at)
```

(In-memory достаточно, Render Free dyno не shared — один процесс. Если будет multi-instance — нужен Postgres/Redis.)

При парсинге edit-intent'а если `task_query` пустой или это «это» / «эту» / «её» / «его» — берём из `LAST_TASK`. Иначе fall-back на `find_task_by_query`.

### 5.7 · Multi-intent split (PR-I3)

«закрой йогу и перенеси отчёт на пятницу» → детектор расщепляет:

1. Сначала зовём `split_message` (уже есть!) → 2 unit'a.
2. На каждый unit зовём `detect_intent`.
3. Выполняем последовательно с собирающим reply: «Закрыл „йогу“ ✅. Перенёс „отчёт“ → пятница.».

### 5.8 · Undo через inline-кнопку (PR-I3)

Каждое подтверждение edit-операции крепит keyboard:

```python
InlineKeyboardMarkup(inline_keyboard=[[
    InlineKeyboardButton(text="↩️ Отменить", callback_data=f"edit:undo:{intent}:{task_id}:{snapshot_id}"),
]])
```

Где `snapshot_id` — id строки в новой таблице `TaskEditSnapshot` со старыми значениями. TTL = 5 минут (потом snapshot gc'ится).

Это **опционально**, можно отложить до PR-I3 — PR-I1/I2 могут жить без undo.

### 5.11 · Промпт для `detect_intent` (черновик)

> Этот промпт пойдёт в `app/ai/prompts/intent.md`. Юзер хочет, чтобы агент НЕ держал его в коде. См. как `reorder.md` устроен.

```
Ты — детектор интента в Telegram-боте планировщика задач.
Юзер пишет голосом или текстом по-русски (часто разговорно).
Твоя задача — понять что юзер хочет сделать с уже существующей задачей,
ИЛИ что он создаёт новую задачу/заметку/реминдер.

Возможные интенты:
- create: юзер создаёт новую задачу/заметку/реминдер. Не интент-команда.
- reorder_horizon: меняет горизонт задачи (сегодня/завтра/неделя/...).
- reorder_time: меняет точное время задачи (с 14:00 на 18:00).
- complete: закрывает задачу как сделанную ("сделал отчёт", "закрой йогу").
- delete: удаляет задачу.
- reopen: переоткрывает закрытую ("верни йогу").
- rename: меняет название задачи.
- set_due: ставит/меняет дедлайн.
- set_priority: меняет приоритет (срочный/обычный/низкий).
- set_category: меняет категорию.
- list_done: показать закрытые сегодня.
- none: непонятно или нет интента к существующей задаче.

Учти:
- Юзер говорит "сделал X" = X.complete (это глагол прошедшего времени).
- Юзер говорит "это" / "эту" / "её" — это анафора к последней созданной/обновлённой задаче. Тогда task_query пустой, агент достанет из контекста.
- "перенеси X на пятницу" — если "пятница" целый день — reorder_horizon (target=week + new_due_raw=пятница). Если "пятница 12:00" — reorder_time.
- "удали" в контексте задачи — delete; "отмени напоминание" — это другая команда (не возвращай delete, верни none).
- "сделай X срочным" — set_priority с new_priority=high.
- "переместил" / "перенеси" / "положи" / "перекинь" / "отнеси" — синонимы reorder/set_category в зависимости от того, куда.
- Если фраза = просто новая задача ("утром пробежка") — intent=create.

Верни JSON по схеме EditIntent.

ПРИМЕРЫ:
[10-15 few-shot примеров, как в reorder.md]
```

### 5.12 · Failure modes и graceful degradation

- **LLM не уверен (`confidence < 0.7`)**: бот переспрашивает «Я понял что ты хочешь закрыть „X“, верно?» + inline `[Да] [Нет]`.
- **Найдено >1 задачи**: см. §5.5.
- **Найдено 0 задач**: «Не нашёл „X“. Может, в архиве?» + `[Показать архив] [Создать новую]`.
- **Задача уже в нужном состоянии** (closed → close): «„X“ уже закрыта.» — без re-event'а.
- **Конкурентный edit с pinned-morning**: после изменения статуса задачи всегда вызываем `refresh_pinned_morning` (уже есть).

### 5.13 · Чего НЕ делаем в PR-I1..I3

- **Аналитика** («сколько закрыл за неделю»). — отдельный PR-J.
- **Habit-tracker** («каждый день пить витамины»). — нужен `Recurrence` модель + scheduler.
- **AI-suggestions** («может, разбить эту задачу?»). — слишком много.
- **Voice-only пайплайн без Whisper-транскрипта**. — Whisper всегда первый шаг.

## 9 · Что строго НЕЛЬЗЯ

- ❌ `git push --force` на main или master.
- ❌ `git commit --no-verify` — pre-commit хуки настроены не зря.
- ❌ `git commit --amend` после push.
- ❌ `git add .` — добавляет лишнее.
- ❌ `getattr` / `setattr` для доступа к атрибутам моделей — это сигнал что ты не понял типы. Используй явные if-маппинги (см. `app/bot/routers/settings.py::_setting_value`).
- ❌ Менять тесты только для того чтобы они проходили. Если тест проверяет старый API surface который PR явно меняет — можно обновить, *с описанием* в комменте теста почему.
- ❌ Динамические Tailwind классы вроде `bg-${tone}-500` — JIT не подхватит. Используй safelist в `tailwind.config.cjs` либо conditional полные классы.
- ❌ Коммитить `.env`, `dist/`, `__pycache__`, `*.pyc`, `node_modules`.
- ❌ Создавать новые SKILL.md без необходимости. У нас уже 21 скилл, скорее обнови существующий через `suggest_skill_pr` чем создавай новый.
- ❌ Дублировать helper'ы. Перед тем как писать новый — `grep` по `app/` сначала.

## 10 · Автономия — **ВАЖНО** (новый раздел, юзер прямо просил)

Юзер сказал дословно:
> «Чтобы следующий бот за первую итерацию прям много чего сделал. Не чтобы по каждому этапу у меня спрашивал, а прям много что сделал. После того как он много чего сделает, он должен это всё сам не спрашивая у меня и не прося у меня запушить, он должен сам это всё обновить, плэнить, документы, запушить, замёржить. Потом написать handoff следующий для следующего агента мне.»

**Правила работы:**

1. **Прочитай этот хендофф полностью + PROGRESS.md (последние 5 записей) + PLAN.md §2 + плана python detailed.md** перед тем как трогать код. Это ~15 минут чтения, экономит часы.

2. **Сразу составь TODO** через `todo_write` — со всеми PR'ами на эту сессию. Желательно minimum 2 PR за итерацию. Например:
   - [ ] PR-I1: complete/delete/reopen голосом.
   - [ ] PR-I2: rename/set_due/set_priority/set_category.
   - [ ] PR-I3: контекст + multi-intent + undo.

3. **Делай и пушь автоматически.** После каждого PR:
   - `uv run pytest -q` (должны быть 0 failures).
   - `uv run ruff format . && uv run ruff check . && uv run mypy app`.
   - `cd webapp && npm run typecheck && npm run build` (только если трогал webapp).
   - Обнови `docs/PROGRESS.md` ВВЕРХУ файла (последний PR всегда первый).
   - `git add <files explicitly>` — никогда `.`.
   - `git commit -m "..."` (без `--no-verify`).
   - `git push` на новую feature-ветку.
   - `git_pr(action="fetch_template")` → `git_pr(action="create")`.
   - **`gh-builtin` запрещает создание PR — используй curl API:**

     ```bash
     curl -sS -X POST -H "Authorization: token ${GUTHUB}" \
         -H "Accept: application/vnd.github+json" \
         https://api.github.com/repos/Itosyro/plan-app/pulls \
         -d '{"title":"...","head":"...","base":"main","body":"..."}'
     ```
   - `git(action="pr_checks", wait_mode="all")` — дождись CI.
   - **Замержь сам** (юзер сказал «я не умею мерджить»):

     ```bash
     curl -sS -X PUT -H "Authorization: token ${GUTHUB}" \
         -H "Accept: application/vnd.github+json" \
         https://api.github.com/repos/Itosyro/plan-app/pulls/<N>/merge \
         -d '{"merge_method":"squash"}'
     ```
   - `git checkout main && git pull` → перед следующим PR.

4. **Когда блокироваться на юзера:**
   - Нужны новые секреты (например OpenRouter keys).
   - Архитектурное решение с реальным trade-off'ом (две одинаково валидные ветки).
   - Что-то опасное (миграция БД дропающая колонку).
   - Финальный отчёт когда всё сделано.

5. **Когда НЕ блокироваться (просто решай сам):**
   - Имя переменной, формат вывода, выбор иконки.
   - Какой intent в EditIntent называть `reorder_horizon` или `change_horizon` — выбирай тот, что ближе к существующему коду.
   - Включать ли DeprecationWarning в этом PR — НЕТ, отдельный PR.
   - Покрывать ли все edge-cases в одном PR — НЕТ, разбивай на I1/I2/I3.

6. **После N PR'ов в сессии — напиши HANDOFF v17.** Обнови:
   - TL;DR (что только что закрыл).
   - §2 (добавь свежие PR-I1/I2/I3 в список phase 6+).
   - §3 GAP-analysis (вычеркни закрытое).
   - §5 если spec изменился по ходу.
   - §10 — добавь свои lessons learned.
   - Файл: `docs/HANDOFF-2026-05-12-v17.md` (или с твоей датой). Закоммить в отдельный PR «docs: handoff v17».

7. **Когда сомневаешься между «спросить» и «сделать»:**
   - Если стоимость ошибки = «откатить PR» — делай.
   - Если стоимость ошибки = «потерять данные / поломать прод» — спрашивай.

8. **Параллельность инструментов.** Всегда читай несколько файлов / запускай несколько independent shell-команд через одну `<function_calls>` секцию. Минимизирует latency.

9. **Используй скиллы.** Перед каждой задачей `skill(action="search", path=".agents/skills", keywords=[...])`. Это твой первый запрос. Не пиши код, не проверив `python-best-practices`, `testing-async-python`, `migrations-safely` и т.д.

10. **Думай 10 шагов вперёд.** Когда добавляешь EditIntent — продумай: как будет работать с multi-intent? Как с undo? Как с conflictющим pinned-morning? Не пиши однопроходное решение, которое потом придётся переписывать.

## 14 · Лессоны из этой сессии (для следующего агента)

- **PROGRESS.md конфликт.** Когда два PR'а параллельно добавляют запись в PROGRESS.md, обязательно будет conflict. Решение — заранее добавлять «свежий PR сверху» и не трогать ничего ниже. См. как разрешён конфликт PR #89: я сохранил подробную версию из feature-ветки и положил её *под* PR-G запись.
- **`git_pr(action="create")` иногда падает с 403** — fallback через `curl` к GitHub API. Работает всегда, токен `${GUTHUB}` имеет полный доступ.
- **`gh` CLI не настроен** в этом окружении — не пытайся. Используй `git_pr` builtin или curl.
- **`webhook_base_url` контролирует keep-alive в локалке.** Если поставишь его в `.env` для локальной разработки — будут пинги на несуществующий host. Оставь `webhook_base_url` пустым локально → property `keepalive_url` вернёт `None` → loop стартует но ничего не пингует.
- **mypy не любит SQLModel join-clauses.** Если в SQL-запросе делаешь `Task.id == TaskEvent.task_id` — это литерал `bool` для mypy. Workaround:

  ```python
  task_id_col = Task.__table__.c.id  # type: ignore[attr-defined]
  event_col = TaskEvent.__table__.c.task_id  # type: ignore[attr-defined]
  ... .join(TaskEvent, event_col == task_id_col) ...
  ```
- **`tasks.py::find_task_by_query`** уже есть, не пиши свой — но он возвращает первую совпавшую. Для PR-I1 расширь до `list[Task]`.
- **Тесты digest'а** используют helper `_onboard(session, ...)` — переиспользуй его для тестов edit-intent'ов. Не пиши свой fixture.
- **Render redeploy after merge** — занимает 1-2 минуты, dyno типа `web` пересоберётся автоматически. Не паникуй если первые 30 сек после merge'а бот не отвечает.
