## 2 · Детали PR-I3

### 2.1 LAST_TASK анафоры
In-memory dict `LAST_TASK: dict[int, tuple[int, float]]` — `{user_id: (task_id, timestamp)}`.
TTL = 60 секунд (monotonic clock).

Обновляется при:
- `persist_classification` — когда создаётся новая Task (не Note).
- `execute_edit` — после успешного выполнения любого executor.
- `cb_edit_resolve` — после disambiguation callback.

Использование: если `intent.task_query` пуст → `pop_last_task(user_id)`. Если найден и TTL не истёк → используется. Иначе → «Не понял, какую задачу ты имеешь в виду.»

### 2.2 PENDING_EDITS для multi-match I2 state
`PENDING_EDITS: dict[int, tuple[EditIntent, float]]` — `{user_id: (intent, timestamp)}`.
TTL = 60 секунд.

При `execute_edit` с >1 match → `store_pending_edit(user_id, intent)`.
В `cb_edit_resolve` → `pop_pending_edit(user_id)` — достаёт полный EditIntent с extra fields (new_title, new_priority и пр.), заменяет task_id и выполняет через `_dispatch_single`.

### 2.3 Multi-intent
После `split_message(text)` каждый unit проходит через `detect_intent(groq_router, unit.text)`:
- edit-интенты → `execute_edit` сразу, reply добавляется в `edit_replies`.
- create-интенты → добавляются в `create_units`, проходят через обычный classify/persist pipeline.
- Если только edit-интенты → возвращается объединённый reply.
- Если смешанные → edit_replies prepend к courier reply.

### 2.4 list_completed_today
Read-only интент `list_done`. Добавлен в `EDIT_INTENTS_I3_READONLY = frozenset({"list_done"})`.
`EDIT_INTENTS_ALL = I1 | I2 | I3_READONLY` — теперь 10 интентов.

Executor `_execute_list_completed_today(user_id)`:
- SELECT Task JOIN TaskEvent WHERE kind='completed' AND created_at >= today_start.
- Возвращает «Сегодня завершено (N): • task1 • task2» или «Сегодня пока ничего не завершено.»

### 2.5 _dispatch_single
Новый хелпер — единый dispatcher для одного task_id. Используется в:
- `execute_edit` (single-match path).
- `execute_edit` (anaphora path).
- `cb_edit_resolve` (disambiguation callback with stored intent).

Заменяет дублированный if/elif chain.

## 7 · Паттерны и best practices

### Session isolation
Каждый executor принимает `(task_id, user_id)`, создаёт `session_scope()`, ищет Task свежую. НИКОГДА не передавать объекты Task между сессиями.

### _dispatch_single
Единый dispatcher — все executors через if/return chain (не elif). Используется в 3 местах: execute_edit, anaphora, cb_edit_resolve.

### E2E test mock sequence
Pipeline call order:
1. `detect_intent(full text)` — intent detection
2. `detect_reorder(text)` — legacy reorder (if create/none)
3. `split_message(text)` — split into units
4. `detect_intent(unit)` × N — **per-unit intent** (PR-I3)
5. `classify_intent(unit)` × N — classify each create-unit
6. `courier_respond(items)` — format reply (if not template_only)

### LAST_TASK
Uses `time.monotonic()` for TTL — immune to system clock changes.

### PENDING_EDITS
Pop-on-use: `pop_pending_edit()` removes the entry. If callback arrives after TTL → None → fallback to simple dispatch (intent_name only, no extra fields).

## 8 · Lessons Learned (PR-I3)

1. **Secret name**: env var is `GITHUB` not `GUTHUB`. v17 had typo.
2. **Multi-intent E2E**: every split unit needs its own `_intent_response("create")` in mock sequence.
3. **_dispatch_single**: eliminates copy-paste if/elif in 3 callsites.
4. **PENDING_EDITS pop semantics**: pop_pending_edit consumes — second pop returns None. Good for one-shot disambiguation.
5. **list_done JOIN**: TaskEvent.kind='completed' + created_at >= today_start gives accurate daily completions.
