# docs/ — как пользоваться этой папкой

5 файлов, у каждого одна тема. Не плодим новые без необходимости — раздувает контекст.

| Файл | Когда обновлять | Кто пишет |
|---|---|---|
| [`PLAN.md`](PLAN.md) | при изменении продуктовых решений (что делаем / не делаем, стиль, сценарии) | человек или AI после согласования с человеком |
| [`ARCHITECTURE.md`](ARCHITECTURE.md) | при изменении технического дизайна (схема БД, AI-пайплайн, ENV-переменные) | AI или человек, **обязательно перед PR**, если PR трогает архитектуру |
| [`ROADMAP.md`](ROADMAP.md) | при сдвиге фаз / содержимого фаз | человек после ревью; AI предлагает |
| [`PROGRESS.md`](PROGRESS.md) | **в каждом PR**, в самом конце, перед коммитом — добавляется новая секция с датой и описанием PR | AI пишет, человек проверяет |
| [`IDEAS.md`](IDEAS.md) | когда появляется идея «когда-нибудь хорошо бы…» или открытый вопрос; и когда юзер сбрасывает поток мыслей | AI или человек |

## Правила, чтобы не сломать

1. **Не дублируй информацию.** Если факт уже есть в `ARCHITECTURE.md` — в `PLAN.md` пиши ссылку, а не копию.
2. **Решения юзера → `PLAN.md` §7** (snapshot ответов). Технические следствия → в `ARCHITECTURE.md` (§2.x, §5.x).
3. **Идеи без решения → `IDEAS.md`**, не в `PLAN.md`. PLAN — только то, что точно делаем.
4. **`PROGRESS.md` — append-only.** Никогда не удаляем старые записи.
5. **Никаких новых .md в `docs/` без явной просьбы.** Если очень нужно — заведи раздел в существующем файле.
6. **Стиль:** русский для содержания, английский для кода/идентификаторов в примерах. Кратко, без воды.
7. **Никаких секретов** (токенов, ключей, паролей) в `docs/`. Никогда. Используй placeholder'ы.

## Что AI делает в начале сессии

1. Прочитай `PLAN.md` целиком (контекст продукта).
2. Прочитай свежие записи `PROGRESS.md` (что было в последних PR).
3. Прочитай `ROADMAP.md` (на какой фазе мы сейчас).
4. Точечно — `ARCHITECTURE.md` по нужной секции.
5. `IDEAS.md` — только если юзер просит «добавь идею» или «найди в идеях».
