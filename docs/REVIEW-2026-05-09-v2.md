## Critical

### R-NEW-C-1 — `/settings` UI cannot change digest times (silently broken)
**File:** `app/bot/routers/settings.py:230-240`
**Severity:** Critical (UI is silently inert; user clicks "08:00" and the bot says "Неверный формат")
**Skill violated:** `defensive-programming/SKILL.md` rule #6 ("Strict callback-data parsing")

The handler uses naive `split(":")`:

```python
parts = callback.data.split(":")
if len(parts) != 4:
    await callback.answer("Неверный формат.")
    return
field = parts[2]
value = parts[3]
```

But the keyboard generator emits values that contain `:` for time settings:

```python
# settings.py:155-167 — _options_keyboard()
callback_data=f"settings:set:{field}:{value}"
# value examples: "07:00", "08:00", "20:00", "21:00", ...
```

So `"settings:set:morning_digest_at:08:00".split(":")` = `["settings", "set", "morning_digest_at", "08", "00"]` (length 5). The `len(parts) != 4` check rejects it as malformed. **Every digest-time option is broken** (4 morning + 4 evening = 8 buttons unusable from the UI).

**How it slipped past tests:** `tests/test_settings.py::test_options_keyboard_has_all_options` only asserts the *button definitions* exist — it never simulates a click and never invokes `cb_settings_set`. The service-layer test (`test_update_user_settings_morning_digest`) calls `update_user_settings(...)` directly, bypassing the broken parser.

**Repro (verified):**
```python
from app.bot.routers.settings import SETTING_OPTIONS
for field, options in SETTING_OPTIONS.items():
    for value, _label in options:
        cb = f"settings:set:{field}:{value}"
        if len(cb.split(":")) != 4:
            print("BROKEN:", field, value, "->", len(cb.split(":")), "parts")
# BROKEN: morning_digest_at 07:00 -> 5 parts
# BROKEN: morning_digest_at 08:00 -> 5 parts
# … (all 8 time values fail)
```

**Fix sketch:** `parts = callback.data.split(":", 3)` (max 3 splits → exactly 4 parts; the value retains its colon). Add a test that monkeypatches `update_user_settings` and asserts every button in every keyboard reaches the service layer with the right `(field, value)` pair.

### R-NEW-C-2 — "сегодня в HH:MM" past now → resolves to next week
**File:** `app/ai/time_resolver.py` (the "push by 7 days when same date and past" branch)
**Severity:** Critical (user says "today at 10", bot quietly schedules for 7 days from now)

When the user writes `сегодня в 10:00` at 14:00 local, `dateparser` returns today 10:00; the resolver detects "past today" and adds 7 days. But the user explicitly used `сегодня` — the right behaviour is to keep the date (so the user sees an immediate "уже прошло" reminder, or the bot offers tomorrow), not silently jump a week.

**Repro (verified):**
```python
from datetime import datetime
from zoneinfo import ZoneInfo
from app.ai.time_resolver import resolve_time
fake_now = datetime(2026, 5, 9, 14, 0, tzinfo=ZoneInfo("Europe/Moscow"))
r = resolve_time("сегодня в 10:00", "Europe/Moscow", now=fake_now)
# resolved_dt = 2026-05-16 10:00 Europe/Moscow  ← +7 days, horizon "week"
```

**Fix sketch:** before applying the +7-day fallback, check the original text: if it contains `сегодня` / `сейчас` / `today`, do NOT roll forward — return the past datetime as-is (caller can decide how to handle "already past"), or roll forward by exactly 1 day (interpreting "сегодня в 10" past 14:00 as "завтра в 10"). Add an explicit test for this fragment + reference time pair.

### R-NEW-C-3 — `delete_task` will fail in Postgres (FK violation, hidden by SQLite tests)
**File:** `app/bot/services/tasks.py::delete_task` + `app/db/models.py::TaskEvent` + all migrations
**Severity:** Critical on production Postgres. SQLite does not enforce FKs by default → tests pass.

`delete_task` writes a `TaskEvent` with `kind="deleted"` referencing the task, then calls `session.delete(task)`:

```python
session.add(TaskEvent(task_id=task.id, kind="deleted", ...))
await session.flush()        # writes TaskEvent referencing task
await session.delete(task)
await session.flush()        # deletes Task → Postgres rejects: FK on task_events.task_id
```

`TaskEvent.task_id` is `ForeignKey("tasks.id")` with default `ON DELETE NO ACTION`. Postgres will raise `IntegrityError` on the second flush; SQLite silently keeps the orphan because PRAGMA `foreign_keys=OFF` by default.

Same issue exists for `Reminder.task_id` (deleted tasks leave orphan reminders that get filtered out by the next-tick JOIN, but rows accumulate forever). Same risk for any FK pointing at `tasks` / `categories` / `horizons` once admin tooling allows their deletion.

**Fix sketch:** decide per-FK whether `ON DELETE CASCADE` or `ON DELETE SET NULL` is correct. For audit trails (`task_events`), CASCADE is right. For `reminders`, CASCADE. For `tasks.source_inbox_id` / `tasks.category_id` / `tasks.horizon_id` — `SET NULL` so a category rename/deletion doesn't drop the task. Two-stage migration (skill rule).

### R-NEW-C-4 — Classifier never sees existing user categories (drives endless duplication)
**File:** `app/bot/routers/_pipeline.py:151`
**Severity:** Critical product behaviour

```python
classify_tasks = [
    classify_intent(groq_router, unit.text, resolved, [], user_tz)
    for unit, resolved in zip(split_result.units, resolved_list, strict=True)
]
```

The 4th argument (`user_categories`) is hard-coded to `[]`. `classify_intent`'s prompt is designed to bias the LLM toward an existing category list — with an empty list, every task gets a freshly-named category. Over a few weeks of use you accumulate "Работа" / "работа" / "Рабочее" / "работ. дела" / "By work" — the entire point of categories is defeated.

**Fix sketch:** fetch `await get_user_categories(session, user_id)` once at the top of `run_pipeline` (already opens a session for resolver context) and pass the list in. Add a test asserting the prompt receives a non-empty `user_categories` for a user who already has tasks.

### R-NEW-C-5 — Webhook idempotency has a TOCTOU race → 500 → Telegram retries forever
**File:** `app/bot/services/inbox.py::is_update_processed` + `record_update` + `app/main.py::telegram_webhook`
**Severity:** Critical (under load — concurrent webhook deliveries from Telegram, or restart-time double-deliveries)

```python
# main.py
async with session_scope() as session:
    if await is_update_processed(session, update.update_id):
        return {"ok": True}
    # … process message …
    await record_update(session, update_id=update.update_id, …)
```

Two concurrent requests for the same `update_id`:
1. Both pass `is_update_processed` (both see "not yet").
2. Both run the handler.
3. Both try to `INSERT INTO telegram_updates` → second one fails with `IntegrityError` on the PK → 500 returned to Telegram → **Telegram retries the webhook indefinitely** (every retry races with itself again). The user also gets duplicate replies.

**Fix sketch:** make the insert idempotent — either `INSERT … ON CONFLICT DO NOTHING` (Postgres-specific) followed by a SELECT, or wrap the INSERT in `try/except IntegrityError` and treat the conflict as "already processed → return 200". Add a test that fires two concurrent webhook requests with the same `update_id` and asserts only one row is inserted and both responses are 200.

The skills already document this exact pattern; it just wasn't applied.

### R-NEW-C-6 — Voice handler ignores user's reminder-offset preset
**File:** `app/bot/routers/voice.py:55-74` (the settings load block)

`handle_text` loads `default_offsets` from `UserSettings.default_reminder_offsets` and forwards it to `run_pipeline`. `handle_voice` does NOT — it only loads `critic_mode`, `courier_*`, and the digest anchors. The pipeline therefore receives `default_reminder_offsets=None` for every voice message, and `_select_reminder_offsets` falls through to the global `DEFAULT_REMINDER_OFFSETS`.

Result: a user who set their reminder preset to "minimal" gets the "default" preset for voice notes. Silently inconsistent with text.

**Fix sketch:** mirror the text-handler block in voice (load `default_offsets` exactly the same way and forward it). Add a regression test.

## Important

### R-NEW-I-1 — Callback handlers crash on malformed `int(parts[N])`
**File:** `app/bot/routers/callbacks.py:126`, `:157`, `:187`, `:204`, `:236`, `:253`, `:284-285`
**Skill violated:** `defensive-programming/SKILL.md` rule #6 (explicit `try/except ValueError`)

Six handlers all do `task_id = int(parts[2])` (and one does `new_category_id = int(parts[3])`) without a `try`/`except`. A malformed callback (e.g., user copy-pastes a stale link, deep-link tampering, future schema change) raises `ValueError` → unhandled in the handler → propagates up to aiogram → 500 → Telegram retries.

**Fix sketch:** add a small helper:
```python
def _parse_int(s: str) -> int | None:
    try:
        return int(s)
    except ValueError:
        return None
```
and use it everywhere. The pattern is mechanical.

### R-NEW-I-2 — `get_or_create_category` / `get_or_create_horizon` race under parallel classify
**File:** `app/bot/services/tasks.py::get_or_create_category` + `get_or_create_horizon`

Inside `_pipeline.run_pipeline` the splitter can produce N units, all classified in parallel via `asyncio.gather`. Each classification result (in `persist_classification`) calls `get_or_create_category` and `get_or_create_horizon`, which currently do read-then-insert. Two concurrent inserts of the same new category/horizon name → one wins, the other raises `IntegrityError` → that unit is dropped from the batch (the `gather` already uses `return_exceptions=True` so the rest survive, but the affected task is silently lost).

**Fix sketch:** either (a) serialise the persistence step (run `persist_classification` sequentially over the survivors — they're at most 5 per message, so the latency cost is irrelevant), or (b) `INSERT … ON CONFLICT (user_id, name) DO NOTHING RETURNING id` then SELECT.

### R-NEW-I-3 — Re-running `/start` post-onboarding crashes
**File:** `app/bot/services/users.py::complete_onboarding`

`complete_onboarding` unconditionally `session.add(UserSettings(user_id=user.id))`. `UserSettings.user_id` is the PK. Re-onboarding (e.g., user wiped their chat and ran `/start` again, or onboarding flow restart after a crash) raises `IntegrityError` and the FSM dies mid-handler.

**Fix sketch:** check if a `UserSettings` row already exists; if so, update its display-name-related fields (none today, but design for it) instead of re-inserting. Reset onboarding fields on the `User` row but keep their settings.

### R-NEW-I-4 — Digests miss the day if the scheduler tick is delayed past the minute boundary
**File:** `app/bot/digest.py::_matches_hhmm` + `tick_digests`
**Skill violated:** `defensive-programming/SKILL.md` rule #7 spirit ("strict HH:MM matching" — the rule is about not slack-matching, but the existing code is too strict in the *other* direction)

`_matches_hhmm` requires `local_dt.hour == hh AND local_dt.minute == mm`. The in-process scheduler ticks every 60 s by default. If a tick fires at 08:00:00 and the next at 08:01:30 (a 90-second drift caused by a Render cold-start, GC pause, or a slow earlier tick), the user's 08:00 digest is permanently skipped that day.

The `last_morning_digest_on` gate is *meant* to handle "fired already today", but the only path that flips the gate is the strict-minute match. There's no "already past 08:00 and we haven't fired yet" path.

**Fix sketch:** change the trigger to "if `local_now.time() >= scheduled_time AND last_*_digest_on != local_now.date()` then fire". The idempotency gate already prevents double-sends. Add tests that simulate a tick at 08:01:30 with `last_morning_digest_on=None`.

### R-NEW-I-5 — `tick_reminders` flush-at-end loses idempotency on crash
**File:** `app/workers/scheduler.py::tick_reminders`

The function loads up to 100 due reminders, sends each one via `bot.send_message`, mutates `reminder.status = "sent"` in memory, then commits at the end of the `session_scope`. If the worker crashes (OOM kill, SIGTERM during deploy) after sending some Telegram messages but before the commit, the next tick sees those reminders still as `pending` and re-sends them.

**Fix sketch:** commit per row (or per small chunk). Or use a "claim" pattern: in a separate transaction up-front, flip status from `pending` to `processing` (so concurrent ticks won't pick the same row), then send and re-flip to `sent`/`failed`. The skill suggests the latter for high-throughput.

### R-NEW-I-6 — `/today` etc. spam Telegram with one message per task
**File:** `app/bot/routers/commands.py::_horizon_handler`

```python
for task in tasks:
    await message.answer(f"{task.title}", reply_markup=task_action_keyboard(task.id))
```

50 tasks → 51 Telegram messages. Telegram's per-chat rate limit is ~30 msgs/sec; a backlog tips into temporary 429s and the bot looks frozen for the user. UX is also poor — the user has to scroll through 50 separate messages just to find one to act on.

**Fix sketch:** paginate (10 per page) with a "next" button, or render a single grouped message and put the action buttons inline with row index. Both approaches drop message count by 5–10×.

### R-NEW-I-7 — All FK constraints have no `ON DELETE` policy
**File:** all `alembic/versions/*.py`
**Severity bumped to Important because data corruption is hidden by SQLite-only tests.**

A grep across the migrations shows zero `ondelete=` arguments. The default `NO ACTION` is the right choice in some cases (audit logs) but the *default* of "do nothing and rely on the app to clean up" creates orphans on Postgres unless every delete path manually deletes children first. The current code does not.

Affected pairs:
* `task_events.task_id → tasks.id` (audit trail): want `CASCADE`.
* `reminders.task_id → tasks.id`: want `CASCADE`.
* `tasks.source_inbox_id → inbox_entries.id`: want `SET NULL`.
* `tasks.category_id → categories.id`: want `SET NULL`.
* `tasks.horizon_id → horizons.id`: want `SET NULL` (or refuse delete).
* `notes.category_id → categories.id`: want `SET NULL`.
* `notes.source_inbox_id → inbox_entries.id`: want `SET NULL`.
* `ai_runs.user_id → users.id`: want `CASCADE` if a user is deleted.
* `ai_runs.inbox_id → inbox_entries.id`: want `CASCADE`.
* `inbox_entries.user_id → users.id`: want `CASCADE`.
* `tasks.user_id → users.id`: want `CASCADE`.
* `categories.user_id → users.id`: want `CASCADE`.
* `horizons.user_id → users.id`: want `CASCADE`.
* `user_settings.user_id → users.id`: want `CASCADE`.
* `telegram_updates.user_id → users.id`: want `SET NULL`.

Two-stage migration (per the skill): add new constraints with the desired policy, then drop the old ones in a follow-up.

### R-NEW-I-8 — No backpressure on background pipelines
**File:** `app/bot/routers/text.py:122` and `voice.py:151`

Each incoming message starts an unbounded background task via `asyncio.create_task`. A user spamming 100 messages spawns 100 concurrent pipelines, each making 3–4 Groq calls. The Groq router has key rotation, but no per-user concurrency cap — the queue grows inside the event loop, blocking the webhook handler if Groq stalls.

**Fix sketch:** an `asyncio.Semaphore` bound (per user, e.g., 3 concurrent pipelines) inside `_pipeline.run_pipeline`, and a global cap (e.g., 50 active pipelines) at the dispatch site.

## Minor

### R-NEW-M-1 — `_setting_value` has unreachable fallback
**File:** `app/bot/routers/settings.py:121`
The `return "—"` fallback can never be hit: `field` is iterated from `SETTING_LABELS`, which exactly matches the explicit if-chain above. Either drop the fallback or convert it to `assert False, "unreachable"` so a future field addition crashes loudly.

### R-NEW-M-2 — `Settings.groq_api_keys` is comma-delimited
If a key ever contains a comma (Groq doesn't today, but service contracts can change), `groq_keys_list` silently corrupts it. Switch to newline-delimited or accept JSON-array env value.

### R-NEW-M-3 — `MAX_VOICE_SIZE` hardcoded
`voice.py:29`: `MAX_VOICE_SIZE = 20 * 1024 * 1024`. Move into `Settings` so we can lower it for free-tier deployments without a deploy.

### R-NEW-M-4 — `voice.py` doesn't reject unknown `file_size`
`if message.voice.file_size and message.voice.file_size > MAX_VOICE_SIZE`. If Telegram omits `file_size` (the field is optional in the Bot API spec), the check is bypassed and `download_file` may pull an arbitrarily large file. Reject `None` too.

### R-NEW-M-5 — `_extract_tg_user_id` doesn't cover all update kinds
`_classify_update` recognises ten update kinds but `_extract_tg_user_id` only inspects `message`, `callback_query`, `edited_message`. Channel posts and inline queries get `tg_user_id=None`, which then ends up in `record_update`. Cosmetic for analytics, but breaks linkability later.

### R-NEW-M-6 — `_select_reminder_offsets` re-`int()`s already-typed values
`int(o)` is redundant given the offsets come from a typed `dict[str, list[int]]`. Drop or replace with `assert isinstance(o, int)` for documentation.

### R-NEW-M-7 — `Reminder` indexes
The schema has separate indexes on `status` and `fire_at`. The hot query is `WHERE status='pending' AND fire_at <= now ORDER BY fire_at`. A composite index `(status, fire_at)` would shrink scan time at scale.

### R-NEW-M-8 — `find_task_by_query` has no relevance scoring
Falls back to "most recently created match". For a user with several similar titles, this is unintuitive. Out of scope for now, but worth noting in IDEAS.md.

### R-NEW-M-9 — Test gaps revealed by the above
* No test simulates concurrent webhook delivery (would catch C-5).
* No test simulates the "delayed scheduler tick" (would catch I-4).
* No test simulates a click on a settings option whose value contains `:` (would catch C-1).
* No test exercises `complete_onboarding` twice for the same user (would catch I-3).
* No test asserts FK behaviour on Postgres (uses SQLite, which silently allows it).

These should be added before any of the corresponding fixes are merged.

## Suggested order of fixes

The dependencies and risk profile suggest the following sequence:

1. **C-5 webhook idempotency race** (the only one Telegram can hammer remotely; cheapest fix).
2. **C-1 settings UI** (one-line fix + test; immediate user-visible win).
3. **C-4 classifier user_categories** (affects every classification; impact compounds).
4. **C-6 voice reminder offsets** (config consistency; one block of code).
5. **C-2 time-resolver "сегодня в HH:MM"** (fixed text fragment check + test).
6. **C-3 + I-7 FK / ON DELETE migration** (one alembic revision, two-stage).
7. **I-1 callback `int()` hardening** (mechanical; safe everywhere).
8. **I-2 category/horizon race** (small surface area).
9. **I-3 re-onboarding** (low priority but easy).
10. **I-4 digest tick robustness** (touches scheduling logic; needs care).
11. **I-5 reminder per-row commit** (medium risk).
12. **I-6 `/today` pagination** (UX polish).
13. **I-8 backpressure** (only matters when traffic ramps up).
14. Minor cleanups (M-1..M-9) bundled into one housekeeping PR.

Each step is independently testable; nothing here requires Phase-5 (Mini App) work.
