### C-2. `Task.due_at` stores **local time** instead of UTC

`dateparser` returns a tz-aware datetime in the user's timezone:

```python
# app/ai/time_resolver.py:128-136
settings = {
    "PREFER_DATES_FROM": "future",
    "TIMEZONE": user_tz,
    "RETURN_AS_TIMEZONE_AWARE": True,
    "RELATIVE_BASE": now.replace(tzinfo=None),
    ...
}
dt = dateparser.parse(preprocessed, languages=["ru"], settings=settings)
```

For a Moscow user saying ┬½ąĘą░ą▓čéčĆą░ ą▓ 15:00┬╗ on 2026-05-09 12:00 UTC,
`dt = 2026-05-10 15:00:00+03:00` (verified locally).

That value is then handed to `persist_classification` and assigned
directly to `Task.due_at`:

```python
# app/bot/services.py:323-330
row: Task | Note = Task(
    ...
    due_at=due_at,
    ...
)
```

`Task.due_at` is a tz-naive `DateTime` column. SQLAlchemy strips the
tz on insert, so the value persisted is `2026-05-10 15:00:00`
**naive ŌĆö but logically MSK, not UTC**. Every other timestamp in the
schema (`created_at`, `received_at`, `processed_at`,
`Reminder.fire_at`, `Reminder.sent_at`, `User.onboarded_at`) stores
**naive UTC** via `utcnow_naive()`.

Today the bug is masked because `Task.due_at` is only used to render
`%H:%M` for display, and the local-time hours happen to be exactly
what the user wants to see. But the schema is now logically
inconsistent and the next time anyone:

* compares `Task.due_at` against `utcnow_naive()` (overdue checks,
  upcoming-task queries),
* exposes `Task.due_at` over a REST API (Phase 5 mini-app ŌĆö the JSON
  consumer will assume UTC),
* migrates to `DateTime(timezone=True)` columns,

ŌĆ”they will silently corrupt data by 3ŌĆō11 hours depending on the
user's tz.

`Reminder.fire_at` is **safe** because `schedule_reminders`
(`services.py:283`) explicitly calls `_to_naive_utc(due_at)` before
subtracting the offset.

* **Fix:** normalise `due_at` to naive UTC at the persistence boundary,
  exactly like `Reminder.fire_at`:
  ```python
  due_at_norm = _to_naive_utc(due_at) if due_at is not None else None
  row = Task(..., due_at=due_at_norm, ...)
  ```
  Then update the two `%H:%M` formatting sites
  (`workers/scheduler.py:33-34`, `bot/digest.py:61-62`,
  `bot/routers/commands.py:58-59`) to convert back to user-local time
  via `_user_local_now`-style helper before formatting. A small
  `format_due_local(task, user_tz) -> str` helper covers all three
  call-sites.
* **Test:** persist a Task with a Moscow-aware `due_at` and assert
  `Task.due_at == expected_naive_utc`; assert
  `format_due_local(task, "Europe/Moscow")` renders the original
  hours.

### C-3. `tick_digests` can double-send if the cron interval drops below 60 s

The digest tick fires once per scheduler interval (default 60 s,
configurable via `SCHEDULER_TICK_INTERVAL_SECONDS`):

```python
# app/bot/digest.py:46-54, 144-157
def _matches_hhmm(local_dt, hhmm):
    return local_dt.hour == hh and local_dt.minute == mm

local = _user_local_now(user.tz, now_utc=now)
morning = _matches_hhmm(local, settings.morning_digest_at)
if morning:
    text = await build_morning_digest(session, user)
    await bot.send_message(chat_id=user.telegram_id, text=text)
```

There is no record of "morning digest already sent today for this
user". The matching window is one wall-clock minute. If the loop ever
ticks faster than 60 seconds ŌĆö for any reason: a future operator
shortens `SCHEDULER_TICK_INTERVAL_SECONDS` for testing, an upstream
change rewrites the loop to back-off-and-retry, or even just
`asyncio.sleep` jitter ŌĆö the same `08:00` minute can match twice and
the user gets two morning digests.

Reminders are protected by `Reminder.status='pending'ŌåÆ'sent'` flips so
the analogous bug can't happen. Digests have no such guard.

* **Fix:** track last-sent date per slot in `UserSettings`
  (`last_morning_digest_on: date | None`,
  `last_evening_digest_on: date | None`) and skip if
  `last_*_digest_on == local.date()`. Set it after a successful send.
* **Test:** call `tick_digests` twice in a row at the same `now`;
  assert the recording bot received exactly one message.

### I-1. `GroqKeyRouter.advance()` is wired up but never called

`app/ai/router.py` was designed to round-robin through a pool of Groq
keys on 429/5xx. The class docstring is explicit:

```python
# app/ai/router.py:3-4
Cycles through a pool of Groq API keys. On 429 or 5xx the caller
should call ``advance()`` to switch to the next key.
```

Yet `rg 'advance\(\)'` over `app/` finds **zero** call-sites outside
the docstring itself. Every Groq client is built once with
`router.current_key` and `instructor`'s built-in `max_retries=2` is
the only resilience layer (`splitter.py:69`, `classifier.py:88`,
`courier.py:149`, `reorder.py:73`, `critic.py:109`).

`max_retries=2` retries on the *same* key. If the operator provides
multiple keys via `GROQ_API_KEYS=k1,k2,k3` (the documented setup ŌĆö
see `.env.example`, `render.yaml`), only `k1` is ever used. Once `k1`
hits its rate-limit window every pipeline call fails until it cools
down.

* **Fix:** wrap each LLM call site with a small helper that catches
  `groq.RateLimitError` / 5xx, calls `router.advance()`, rebuilds the
  client, and retries up to `len(keys)` times. Or move that logic
  inside `GroqKeyRouter.with_retry(...)` so call-sites stay terse.
* **Test:** mock two respx responses (429 then 200) for the same
  `chat/completions` URL with different `Authorization` headers; assert
  the second request used `k2`.
