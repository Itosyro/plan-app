## 1 · NEW critical (must-fix before next public release)

### R-NEW-3-C-1 — Splitter prompt over-eagerly fragments single intents on the 8B model
**Files:** `app/ai/prompts/splitter.md`, `app/ai/splitter.py:24` (`SPLITTER_MODEL = "llama-3.1-8b-instant"`).
**Severity:** **Critical** (single user-visible failure case the user explicitly complained about — "поставь напоминание на футбол в 7 → каждое слово как отдельная задача").
**Repro path:** `tests/test_splitter.py` covers only happy multi-intent cases (`"купить хлеб и молоко, а ещё записаться к врачу"`); there is **no test that asserts a single-clause reminder is preserved as 1 unit**.

**Diagnosis:**
1. Current prompt has 4 examples, all of which split. There is no negative example like `"поставь напоминание о том что вечером у меня футбол в 7" → 1 unit`. With temperature 0.0 + a small instruction-tuned model, the prompt's distribution is biased toward "find separators".
2. The prompt does not name the failure mode (`"DO NOT split if the message is a single instruction even if it contains commas, prepositions or subordinate clauses"`).
3. `llama-3.1-8b-instant` is the cheapest Groq model — it routinely loses the antecedent of a long Russian sentence with embedded subclauses.
4. There is no holistic signal back to the classifier: each unit is classified in isolation (`_pipeline.py:279-281`), so even when the splitter is wrong, the classifier can't recover ("everything is one reminder").

**Recommended fix path** (PR-F, after measurement):
- Add ≥6 negative examples to the prompt (`"single intent — keep as 1 unit"`).
- Add an explicit rubric: *"a `напомни …` / `напоминание о …` clause is always 1 unit, regardless of internal commas"*.
- Add a regression test: `test_splitter_keeps_reminder_intact` parametrised over 8–10 phrases.
- If still flaky, introduce **multi-provider routing** (per user's plan): swap `SPLITTER_MODEL` to a more capable model (DeepSeek-V3 / MiniMax-M2 via OpenRouter) for the splitter stage only, keep classifier on Groq 70B. Provider abstraction is described in §5.

**Impact when fixed:** Removes the most-noticed UX wart of the current bot.

### R-NEW-3-C-2 — `Reminder` rows are silently de-duplicated by `(due_at, user_id)` index but the reminder pipeline doesn't handle the IntegrityError
**Files:** `app/db/models.py` (Reminder lines 256+ — see `alembic/versions/2026_05_08_2015-0003_phase_4_reminders.py`), `app/bot/services/tasks.py::persist_classification`.
**Severity:** **Important** (in heavy concurrent webhook traffic the same reminder can be persisted twice from two retries; the second one will collide on the unique index and the entire batch transaction rolls back, **losing the surrounding tasks**).

**Diagnosis:** Persistence is done in a single `session_scope()` block (`_pipeline.py:315-361`). One bad row poisons the whole flush.

**Recommended fix:** Wrap each `persist_classification` in its own subtransaction (`session.begin_nested()`) or use `INSERT ... ON CONFLICT DO NOTHING` on the reminder insert specifically. Adopt the same pattern as `get_or_create_category`.

**Tag:** Should land alongside PR-D (soft-delete + cleanup) since both touch reminder lifecycle.

## 4 · NLU diagnosis (specific to user's complaint)

The user described the bot's biggest practical failure: *one voice message, one intent ("поставь напоминание о футболе в 7"), and the bot creates several disjoint tasks.*

**Where it goes wrong** (in pipeline order):

1. **Splitter** (`app/ai/splitter.py`, model `llama-3.1-8b-instant`):
   - Prompt has 4 examples, all positive-split. No negative examples. **Fix in prompt.**
   - 8B is small for Russian sub-clauses. **Fix possibly in model swap (DeepSeek-V3 free / paid).**

2. **Time resolver** (`app/ai/time_resolver.py`, pure-Python, 242 LOC):
   - This is rule-based and sees only one unit at a time. If the splitter already mis-fragmented, it can't recover.

3. **Classifier** (`app/ai/classifier.py`, `llama-3.3-70b-versatile`):
   - 70B is fine here for Russian. The classifier is given the unit + categories + timestamp. The prompt has no instruction to look at neighbours / set membership. So if the splitter is wrong, the classifier independently mis-classifies each fragment.

4. **Critic** (`app/ai/critic.py`, `qwen-qwq-32b`):
   - Runs only when classifier confidence < 0.7 (default). For confidently-wrong fragments (8B splitter spits a confident "позвонить маме" fragment that was actually part of a longer reminder), critic doesn't fire.

5. **Persist** (`_pipeline.py::_run_pipeline_inner`):
   - Each survivor becomes its own row. **No de-duplication step.**

**Conclusion:** The root cause is at stage 1 (splitter prompt + model). Fixing it in isolation will save 80% of the user-reported failures. Stage 4 (critic) needs a "duplicate-check" stage to catch the residual 20%.

**Recommended sequence:**
- Phase A (PR-F1, low risk): rewrite the splitter prompt with ≥6 negative examples and a regression-test fixture in `tests/test_splitter.py`. Re-measure on a hand-curated Russian dataset of 30–50 messages. Report the % improvement.
- Phase B (PR-F2): if Phase A leaves >10% failures, swap splitter to OpenRouter DeepSeek-V3. **Cost-permitting** — the multi-account-rotation strategy the user described is what enables this.
- Phase C (PR-F3): add the multi-stage critic chain (§3 / R-NEW-3-A-2) with the dedup stage at the end.

## 5 · Provider abstraction — design sketch (input for PR-F)

User confirmed strategy: **multiple OpenRouter accounts in rotation**, plus tiered critic.

**Risks the agent must accept** (already disclosed to user):
- OpenRouter ToS forbids multi-account farming. Accept that bans are possible and engineer a **graceful-degradation fallback** to Groq.
- Free OpenRouter models (DeepSeek-V3, MiniMax-M2 free tier) have aggressive 429s — rotation must be fast (sub-second) and the global limit per app instance must be tuned (~1 request / second / account).

**Design:**
```
LLMProvider (abstract):
  async def chat(self, *, model, messages, response_model, temperature) -> T
  is_recoverable_error(exc) -> bool

LLMKeyRouter[P: LLMProvider]:
  keys: list[str]
  provider: P
  current_key, advance, async_client (provider-specific)

call_with_rotation[P, T](router: LLMKeyRouter[P], fn)
  # unchanged logic, just generic over provider
```

**Concrete providers:**
- `GroqProvider` (existing — wraps `AsyncGroq`).
- `OpenRouterProvider` (new — wraps `openai.AsyncOpenAI(base_url="https://openrouter.ai/api/v1")`).

**Per-stage routing:**
```python
# app/shared/config.py
SPLITTER_ROUTING: list[ProviderSpec]      # try in order; on exhaust, fall through
CLASSIFIER_ROUTING: list[ProviderSpec]
CRITIC_ROUTING: list[ProviderSpec]
COURIER_ROUTING: list[ProviderSpec]
```

Each `ProviderSpec` is `(provider, model, env_keys_var)`. So the user can declare in env:
```
PLAN_LLM_SPLITTER_ROUTING=openrouter:deepseek/deepseek-chat-v3:OPENROUTER_KEYS_A,groq:llama-3.1-8b-instant:GROQ_KEYS
```
…and the bot tries OpenRouter first, falls back to Groq on exhaustion.

**Multi-stage critic:** see §3 / R-NEW-3-A-2. Each stage gets its own `ProviderSpec`, so the user can pin the dedup-stage to a tiny-cheap model and the semantic-stage to a stronger one.
